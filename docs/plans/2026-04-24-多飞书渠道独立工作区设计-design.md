# 多飞书渠道独立工作区设计

## 背景

当前系统已经具备统一消息抽象、Gateway 多通道入口、会话归档和可配置的 LLM Provider，但它的真实运行模型仍然是“单个 AgentApplication + 单个 workspace + 单个飞书 adapter”。这意味着虽然代码里出现了 `channel`、`adapters` 这类多渠道概念，实际可运行形态仍然只有一套运行时状态：`SOUL.md`、`MEMORY.md`、`skills/`、`sessions.db`、Heartbeat/Cron 配置和 M-flow 数据都绑定到一个 workspace。对于单 bot 或单 Web 控制台，这个结构足够简单；但只要进入多飞书机器人场景，状态和路由都会立刻变得不安全。

本次需求的目标不是简单支持“多个飞书连接”，而是支持“多个相互独立的飞书渠道实例”，每个实例拥有自己的 workspace、技能目录、记忆文件、会话归档和运行上下文。也就是说，我们需要从“多渠道类型”升级为“多运行时实例”。这样不同飞书机器人可以共用同一个进程和代码库，但互不串会话、互不串技能、互不串长期记忆。

## 当前架构约束

当前约束主要集中在三层。第一层是 Gateway：`GatewayServer.adapters` 目前是按渠道类型注册的，飞书适配器被写死为 `"feishu"`，因此同进程中只能有一个飞书 bot。第二层是会话主键：`CommandQueue`、`SessionManager` 和 `SessionArchive` 都只认 `session_id`，而飞书消息里当前直接把 `chat_id` 当作会话键，这会让不同 bot 的同一 chat 标识无法隔离。第三层是运行时装配：`build_app_async()` 只会用 `agent.workspace_dir` 构造一个应用实例，所有会话数据、记忆文件和技能目录都落在一个 workspace。

这三个约束叠加后，当前系统不适合直接“多开几个飞书 adapter”来解决问题。即使勉强接入多个 bot，也会遇到至少四类问题：队列 lane 冲突、Session 内存缓存冲突、SQLite 主键冲突，以及恢复历史时丢失渠道实例上下文。换句话说，现有架构的瓶颈不在飞书 SDK，而在内部状态模型仍然只有单 runtime。

## 方案对比

### 方案 A：仅扩展会话键

最小做法是保留单个 `AgentApplication`，只把 Feishu 的 `session_id` 改造成命名空间形式，例如 `feishu:bot_a:chat_id`。这样可以在不大动 Gateway 的前提下缓解队列和归档冲突。优点是实现成本低；缺点是所有 bot 仍共享同一套 workspace、skills、SOUL/MEMORY、Heartbeat/Cron 和 M-flow 数据。这只能解决“线程不撞”，不能解决“状态独立”。

### 方案 B：单进程多 runtime

推荐方案是在单进程内引入多个 runtime。每个 runtime 对应一个渠道实例，包含：一个 `AgentApplication`、一个 workspace、一个 adapter、一个 runtime 标识和一套会话键命名空间。Gateway 负责按 runtime 路由消息与响应。优点是隔离清晰、资源复用好、部署仍然简单；缺点是 Gateway 和配置模型需要升级，调度器需要分阶段适配。

### 方案 C：多进程多 bot

另一条路线是每个飞书机器人单独起一个进程，各自使用不同配置目录和 workspace。优点是隔离最强、实现直接；缺点是部署与监控成本高，不利于后续共享管理能力，也无法在一个 Gateway 内集中管理多渠道实例。对于当前项目目标来说，这条路线更像运维策略，而不是产品架构。

## 推荐方案

本次采用方案 B，并分两阶段实施。Phase 1 先让 Gateway 支持多个 Feishu runtime，每个 runtime 拥有独立 workspace、独立会话命名空间和独立 AgentApplication。这样就能满足当前“多飞书机器人独立渠道”的核心诉求。Phase 2 再把 Heartbeat、Cron 和主动推送从“单 workspace/单 feishu”扩展为“按 runtime 调度”。这种拆法的好处是能先把最关键的收发链路和状态隔离做对，再逐步处理调度器这类外围能力，避免首轮实现范围失控。

## Phase 1 架构设计

### 1. 运行时配置模型

在现有 `agent/providers` 配置之外，新增一个可选的 `channels.instances` 配置段。每个实例描述一个独立 runtime，至少包含：

- `name`: 运行时唯一标识，例如 `feishu-main`
- `type`: 渠道类型，Phase 1 固定为 `feishu`
- `workspace_dir`: 独立工作区路径
- `app_id_env` / `app_secret_env`: 凭证环境变量名

如果 `channels.instances` 缺失，则系统保持当前单 runtime 行为，继续使用 `agent.workspace_dir`。这保证 CLI、Web 和既有配置文件都能无缝兼容。多 runtime 模式下，Gateway 只消费 `channels.instances`，每个实例都通过同一套 provider 配置装配自己的 `AgentApplication`。

### 2. 消息与线程标识

`NormalizedMessage` 新增 `channel_instance` 字段，用来标识消息属于哪个具体渠道实例，例如 `feishu-main`。同时引入内部线程键概念：`thread_key = f\"{channel}:{channel_instance}:{session_id}\"`。这里的 `session_id` 继续保留外部会话标识语义，例如飞书的 `chat_id`，用于真正回消息；而 `thread_key` 专门用于内部排队、会话恢复和归档。这样一来，回复层继续拿原始 chat_id 发消息，状态层则使用稳定且无冲突的复合主键。

### 3. Gateway 路由

`GatewayServer` 从“单 app + 多 adapter”升级为“多 runtime app + 多 adapter 实例”。适配器注册不再固定使用 `"feishu"` 作为键，而是使用实例名，例如 `feishu-main`、`feishu-ops`。Gateway 维护两张表：

- `runtime_apps[runtime_id] = AgentApplication`
- `adapters[adapter_id] = ChannelAdapter`

同时保留单 runtime 兼容模式：旧代码和旧测试仍然可以继续使用 `"feishu"` 作为默认 adapter 键。当 Gateway 收到消息时，通过 `message.channel_instance` 找到对应 runtime，再调用该 runtime 的 `app.core.run()` 或 `run_events()`。响应发送同理，通过 `channel + channel_instance` 定位 adapter。

### 4. 独立 workspace

每个 runtime 在构建应用时都使用自己的 `workspace_dir`。这样 `SOUL.md`、`MEMORY.md`、`skills/`、`sessions.db`、日志和后续 M-flow 数据都会随 runtime 分开。为了避免与当前入口层 `Path(\"workspace\")` 的硬编码冲突，Phase 1 中 Gateway 的多 runtime 模式只保证 AgentApplication 级别的数据隔离；调度器、统一日志目录等仍保留单实例行为，并在启动时输出明确警告。

## 错误处理与兼容性

多 runtime 模式下最重要的错误处理是配置校验与启动降级。对于每个渠道实例，需要在启动时校验：

- `name` 唯一
- `workspace_dir` 可创建
- 环境变量名存在且非空
- `type` 为当前支持的渠道类型

如果某个实例启动失败，不应拖垮其他实例；Gateway 可以记录错误并继续注册可用实例。如果最终没有任何实例可用，则启动失败。兼容性上，旧配置文件无需修改；只要不声明 `channels.instances`，系统仍按当前单 runtime 工作。

## 测试策略

Phase 1 的测试重点不是飞书 SDK 本身，而是新的路由和隔离语义。至少需要补上四组测试：

1. 配置加载：能解析多个渠道实例，且保持旧配置兼容。
2. 消息线程键：同一 chat_id 在不同 runtime 下产生不同 `thread_key`。
3. Gateway 路由：两个 Feishu adapter 各自把消息路由到自己的 `AgentApplication`。
4. 会话恢复：`SessionManager` / `SessionArchive` 使用 `thread_key` 后，跨 runtime 不串历史。

Heartbeat/Cron 的多 runtime 调度不纳入本次实现闭环，但需要在设计上预留 `channel_instance` 和 runtime workspace 的扩展位，避免二次重构。
